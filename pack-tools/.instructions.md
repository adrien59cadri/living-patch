# Pack Creation & Review Instructions

**Quick reference**: See root `context.md` for project overview. This file is pack-specific.

You are assisting with the creation, review, and generation of LivingPatch data packs. Use these guidelines when:
- Generating new data packs (e.g., "Create a pack for Pacific Northwest birds")
- Reviewing existing packs for correctness and completeness
- Suggesting improvements to pack structure or content
- Adding new species relationships to packs

## Pack Structure Overview

Each pack is a JSON file with two main sections:

### Metadata (Required)
Describes the pack itself:
```json
{
  "metadata": {
    "id": "pack-unique-id",           // lowercase, hyphens/underscores only
    "createdDate": "2026-05-26T00:00:00Z",  // ISO 8601 format
    "author": "Your Name",
    "version": "1.0.0",                // Semantic versioning
    "schemaVersion": "1.0.0",          // Schema version
    "description": "What's in this pack",
    "status": "published"               // Optional: "published" (default) or "draft"
  }
}
```

### Data (Required)
Contains species, relationships, and metadata:
```json
{
  "data": {
    "species": [],          // Individual species and taxonomic groups
    "taxonomic_groups": [], // Optional separate group array
    "symbiosis": [],        // Symbiotic relationships
    "relations": []         // General relations
  }
}
```

## Species Entry Guidelines

Each species entry has **required** and **optional** fields.

### Required Fields (Always Include)

| Field | Purpose | Example |
|-------|---------|---------|
| `id` | Unique identifier | `bird_pileated-woodpecker` |
| `common_name` | Display name | `Pileated Woodpecker` |
| `form` | Species category | `bird`, `mammal`, `tree`, `wildflower` |
| `habitat` | Where it lives | `["forest", "woodland"]` |
| `diet` | What it eats | `["insect_eater"]` or `["herbivore"]` |
| `behavior` | Key behaviors | `["cavity_excavator", "nester"]` |
| `season` | When active | `["year_round"]` or `["spring", "summer"]` |
| `functional_description` | Narrative description | "Largest woodpecker in PA..." |
| `life_stages` | Life cycle info | `[]` or array of stage objects |
| `region` | Geographic region | `northeast_pa` |

### Optional But Important Fields

| Field | Purpose | Example |
|-------|---------|---------|
| `latin_name` | Scientific name | `Dryocopus pileatus` |
| `ecological_role` | Trophic role | `carnivore`, `herbivore`, `producer`, `pollinator` |
| `is_keystone` | Is it keystone? | `true` |
| `keystone_type` | Type of keystone role | `ecosystem_engineer`, `predator`, `host_plant` |
| `keystone_description` | Why it's keystone | "Primary cavity excavator..." |
| `active_months` | Active months | `["May-Sep"]` |
| `taxonomic_group` | Parent group ID | `group_woodpeckers` |
| `is_group` | Is this a group? | `true` for taxonomic groups |

### ID Format Rules

**Species IDs** follow format: `{category}_{slug}`

Examples:
- `bird_pileated-woodpecker` ✓ (bird category, multi-word slug)
- `mammal_eastern-gray-squirrel` ✓ (mammal category, multi-word slug)
- `plant_white-oak` ✓ (plant category, multi-word slug)
- `insect_monarch-butterfly` ✓ (insect category, multi-word slug)

Invalid formats:
- `pileated-woodpecker` ✗ (missing category prefix)
- `bird_piliated_woodpecker` ✗ (underscores in slug instead of hyphens)
- `BIRD_WOODPECKER` ✗ (uppercase)

**Categories to use:**
- `bird`, `mammal`, `plant`, `tree`, `wildflower`, `shrub`, `insect`, `butterfly`, `beetle`, `bug`, `bee`, `amphibian`, `frog`

When species has distinct life stages, describe them:

## Relationships Guidelines

### Status Field

The optional `status` field controls when a pack is loaded:

- **`"published"`** (default): Reviewed and approved. Always loaded by the app.
- **`"draft"`**: Work-in-progress. Only loaded with `--include-drafts` flag.

Use `"draft"` status for:
- Packs under peer review
- Experimental species/relationships
- Community contributions pending approval
- Development and testing

Once reviewed and approved, change status to `"published"` to include in app distribution.

## Symbiosis (Most Important)

Describes ecological relationships between species:

```json
{
  "type": "mutualism|parasitism|predation|competition|commensalism",
  "members": ["species_id_1", "species_id_2"],
  "impacted_species": "species_id_1",  // Optional: who benefits/suffers
  "obligate": true|false,               // Is relationship required?
  "notes": "Detailed description of how they interact"
}
```

### Types:
- **mutualism**: Both species benefit (e.g., pollinator ↔ plant)
- **parasitism**: One benefits, one harmed (e.g., caterpillar eats plant)
- **predation**: One eats the other
- **competition**: Both compete for same resource
- **commensalism**: One benefits, one unaffected

### Examples:

```json
{
  "type": "parasitism",
  "members": ["insect_monarch-butterfly", "plant_common-milkweed"],
  "impacted_species": "insect_monarch-butterfly",
  "obligate": true,
  "notes": "Monarch caterpillars are obligate on milkweed — cannot breed without Asclepias genus."
}
```

```json
{
  "type": "mutualism",
  "members": ["insect_native-bumble-bees", "plant_common-milkweed"],
  "notes": "Bumble bees are primary diurnal pollinators of milkweed — produce 8x more mature pods than nocturnal moths."
}
```

### Validation Requirements:
- Both `members` IDs must exist in the pack or base dataset
- `impacted_species` (if provided) must be in `members`
- `obligate` should reflect biological reality
- `notes` must be non-empty and informative

## Common Relationships to Model

### Predation Networks
- Predator → Prey relationships
- Food web connections

### Host-Plant Relationships
- Herbivorous insects → Plant hosts
- Must be obligate if insect is specialist

### Pollination Mutualisms
- Pollinator → Flower relationships
- Note which are primary vs. secondary pollinators

### Cavity Cascades
- Primary cavity excavators (e.g., Pileated Woodpecker)
- Secondary cavity users (e.g., owls, bluebirds)

### Frugivore-Plant Relationships
- Frugivore → Fruit-producing plant
- Describes seed dispersal

## Validation Checklist

Before submitting a pack, verify:

- [ ] All species IDs match format `category_slug` (lowercase, hyphens)
- [ ] No duplicate IDs within pack
- [ ] All required fields present in each species entry
- [ ] All symbiosis/relation member IDs exist (in this pack or base data)
- [ ] No orphaned references
- [ ] Dates are ISO 8601 format
- [ ] Version is semantic versioning (e.g., 1.0.0)
- [ ] Pack ID is lowercase with hyphens/underscores only
- [ ] Description is non-empty and clear
- [ ] Latin names are correct where provided
- [ ] Relationships make ecological sense
- [ ] Keystone descriptions explain *why* species is keystone

## Review Best Practices

When reviewing a submitted pack:

1. **Validate schema**: Run `npm run validate packs/pack-name.json`
2. **Check conflicts**: Run `npm run merge packs/0-base.json packs/pack-name.json --strict`
3. **Verify accuracy**: Cross-check species descriptions and relationships with sources
4. **Check completeness**: Ensure important relationships are documented
5. **Test merge**: Confirm pack integrates cleanly with base dataset
6. **Ecology check**: Ensure relationships reflect real ecological dynamics
7. **Documentation**: Verify metadata is clear and complete

## Examples to Reference

Look at `packs/0-base.json` for comprehensive examples of:
- Well-structured species entries
- Diverse relationship types
- Keystone species descriptions
- Life stage documentation
- Taxonomic group definitions

## Quick Checklist for Generation

When asked to "Create a pack for [topic]":

1. ✓ Create metadata with valid id, author, version, dates
2. ✓ Define species with all required fields + relevant optional ones
3. ✓ Create taxonomic group entry if multiple related species
4. ✓ Document symbiosis/predation/mutualism relationships
5. ✓ Include keystone status and descriptions where applicable
6. ✓ Validate format (schema compliance, ID uniqueness, references)
7. ✓ Test merge with base dataset
8. ✓ Return complete, valid JSON

## Common Errors to Avoid

| Error | Example | Fix |
|-------|---------|-----|
| Invalid ID format | `pileated_woodpecker` | Use `bird_pileated-woodpecker` |
| Orphaned reference | Symbiosis references `bird_unknown` | Add species or fix ID |
| Missing required field | No `functional_description` | Add description |
| Duplicate IDs | Two entries with same ID | Rename or remove one |
| Wrong date format | `5/26/2026` | Use `2026-05-26T00:00:00Z` |
| Bad semver | `1.0` | Use `1.0.0` |
| Obligate relationship mismatch | Parasitism marked `obligate: false` if truly required | Correct to match biology |

## Questions to Ask When Adding Relationships

1. **Do these species actually interact?** If no, don't create fake relationships.
2. **What type of interaction?** Is it predation, competition, mutualism, etc.?
3. **Is it obligate?** Does one species absolutely require the other?
4. **Why does it matter?** Write notes explaining ecological significance.
5. **Which direction?** Who benefits or is affected? Set `impacted_species`.

# Using Claude Code to Generate & Correct Packs

This section describes how to use Claude Code (or Copilot) to generate and correct ecological data packs. Claude can convert free-form notes, CSVs, or descriptions into valid LivingPatch pack JSON.

## When to Use Claude Code for Packs

**Best use cases:**
- Converting CSV species lists into structured pack entries
- Adding missing species from field notes or descriptions
- Generating symbiosis entries from narrative descriptions
- Correcting broken references in existing packs
- Enriching incomplete species entries
- Creating new packs for different regions or topics

## How to Get Started

### 1. Read the Pack Schema Guide

Before asking Claude to generate a pack, reference:
```
pack-tools/PACK-CONTEXT.md
```

This document has:
- Complete pack structure with examples
- ID format rules (critical!)
- Field descriptions and validation rules
- Common mistakes and fixes
- Tips for generation

### 2. Open Claude Code

- **Web**: Visit `claude.ai/code` and clone this repo
- **CLI**: Run `claude dev` in the repo root
- **GitHub**: Use Claude Code via "Ask Claude" in GitHub UI

### 3. Provide Input to Claude

Claude can work with any of these input formats:

**Option A: Free-form description**
> "Add Eastern Tiger Swallowtail butterfly. Caterpillars eat Black Cherry and oaks. Adults pollinate in spring/summer. Migratory."

**Option B: CSV or tabular data**
```csv
common_name,latin_name,form,habitat,diet
Carolina Wren,Thryothorus ludovicianus,songbird,"forest_edge, garden",insect_eater
```

**Option C: Existing pack with errors**
Paste broken pack JSON and ask: "Fix broken symbiosis references in this pack."

**Option D: Narrative from field notes**
Paste researcher notes about species and ask: "Convert these species observations into a pack."

## Best Practices

### 1. Always Reference PACK-CONTEXT.md

Include in your Claude prompt:
```
Reference pack-tools/PACK-CONTEXT.md for:
- ID format rules (category_slug pattern)
- Required vs optional fields
- Symbiosis type definitions
- Validation rules
```

Claude needs these rules to generate valid JSON consistently.

### 2. Be Specific About Symbiosis Type

Don't just say "add relationships." Specify:
- "Add mutualism between bumble bee and milkweed"
- "Add obligate parasitism: Monarch caterpillar on milkweed"
- "Add commensalism: Wren uses woodpecker cavity"

This helps Claude pick the right relationship type.

### 3. Validate After Generation

After Claude generates a pack, ask it to:
1. **Check ID format**: "Verify all species IDs match `category_slug` pattern"
2. **Check references**: "Verify all symbiosis member IDs exist in the species array"
3. **Check required fields**: "Ensure every species has common_name, functional_description, and life_stages"

### 4. Run Pack Validation

After getting JSON from Claude, validate locally:
```bash
npm run pack:validate packs/your-pack.json
```

If errors appear, paste them back to Claude: "Fix these validation errors: [errors]"

### 5. Review Symbiosis Carefully

Claude sometimes generates relationships that *sound* right but aren't ecologically accurate. Always review:
- Does this relationship actually exist in nature?
- Is the type correct (mutualism vs parasitism)?
- Should obligate be true/false?

## Tips for Better Results

### Tip 1: Provide Context
```
GOOD: "Add Great Blue Heron. It's an apex aquatic predator in PA wetlands, 
hunts fish/frogs, year-round resident. Make it keystone."

LESS GOOD: "Add a bird that lives in water"
```

### Tip 2: Specify Active Months
```
GOOD: "Eastern Tent Caterpillar is active April-May only"

LESS GOOD: "Caterpillar that appears in spring"
```

### Tip 3: Link to Existing Species
```
GOOD: "Carolina Wren nests in cavities made by Pileated Woodpecker.
Generate symbiosis: type=mutualism, members=[bird_carolina-wren, 
bird_pileated-woodpecker], obligate=false"

LESS GOOD: "Add a relationship between wren and woodpecker"
```

### Tip 4: Ask for Validation
After Claude generates, always ask:
```
"Validate this pack:
1. All species IDs match category_slug format ✓
2. All symbiosis members exist in species ✓
3. All required fields present ✓
4. Relationships are ecologically accurate ✓
5. Keystone descriptions explain why ✓"
```
