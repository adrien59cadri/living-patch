# LivingPatch Development Instructions

**Project**: Personal ecological literacy tool for nature hobbyists in NE Pennsylvania.
**MVP Phase**: Read-only dataset explorer (local webapp, no server).
**Tech**: React 18 + TypeScript + Vite + TailwindCSS v4 + React Router v6.

## Quick Start
```bash
cd app
npm install  # If needed
npm run dev  # http://localhost:5173
```

✅ Full-text search (common name, latin name, description, all tags)
✅ Multi-select filters (form, season, habitat; combine with search)
✅ Species list with keystone badges
✅ Species detail card with:
  - Photo placeholder
  - Latin name (collapsible)

## How To...

### Search and filter
- User types in search box (debounced 200ms)
- Full-text across: common_name, latin_name, functional_description, habitat, diet, behavior, season, form
- Click filter chips to toggle active filters
- Filters combine: "bird" AND "summer" returns summer-active birds
- "Clear filters" button resets all filters but keeps search

### View species relationships
- Click any species in list → DetailPage
- Scroll to "Ecological Relationships" section
- Grouped by type: Mutualism, Parasitism & Hosting, Predation, Commensalism, Competition, Related Species
- Obligate relationships pinned at top (e.g., Monarch → Common Milkweed marked obligate)
- Each related species is a clickable tile (unless it's a group)
- Max 6 tiles per section; "Show more" expands overflow

### Add a new species
1. Add entry to `livingpatch-mvp-dataset.json` (and sync to `app/src/data/dataset.json`)
2. Ensure `id` follows format: `type_common-name-slug`
3. Run `npm run dev` (data reloads on save)
4. Search/filter will find the new species immediately

### Add a new symbiosis relationship
1. Add entry to `symbiosis[]` array in dataset JSON
2. Reference existing species IDs in `members[]`
3. Set `obligate: true` if required relationship
4. DetailPage will show it in appropriate grouped section

### Change styling
1. Edit component `.tsx` files — add/modify Tailwind classes
2. Edit `src/index.css` only if adding global styles (rare)
3. Color palette: stone (neutrals), emerald (primary), amber (keystone/highlights)
4. Mobile-first: use `sm:`, `lg:` breakpoints as needed

### Debug
- Open browser DevTools → Console (should be empty)
- Check Network tab (should have zero requests after initial load)
- Offline mode (DevTools → Network → offline): app should work fully
- Search for console errors if behavior is unexpected

## Code Principles
1. **Types**: Strict TypeScript; no `any` types; interfaces in `src/types/index.ts`
2. **Data**: Static JSON, indexed into Maps for O(1) lookups; never refetch
3. **Components**: Functional React with hooks; <300 LOC per file
4. **Testing**: Manual E2E in browser for now (test search, filters, navigation, offline)
5. **Naming**: Domain-specific (SpeciesCard not DetailView, RelationshipTile not LinkCard)

## Performance Notes
- Data loaded once at startup; O(1) lookups via Map objects
- `filterSpecies()` is O(n) but acceptable for 30–40 species
- `useDataset()` hook exposes pre-indexed data; use it everywhere
- `useMemo()` wraps filtered results to avoid re-filtering on every render
- No infinite loops or circular dependencies

## Accessibility
- Buttons have `aria-label` if icon-only
- Links have descriptive text (not "click here")
- Semantic HTML: `<header>`, `<main>`, `<button>`, `<a>` tags
- Keyboard navigation: Tab through buttons, Enter to activate
- Color not the only signal (use icons, text, borders)

## Deployment (Future)
- `npm run build` → `dist/`
- Host on static host (Netlify, Verge, GitHub Pages)
- Works from `file://` URLs too (HashRouter + no server dependency)
- Consider PWA manifest for install-to-home-screen

## What to Avoid
- ❌ `import * as` from large modules (explicit imports)
- ❌ Inline styles (use Tailwind classes)
- ❌ Global state (use React hooks for local state only; Phase 2 will use Zustand)
- ❌ External API calls (ever)
- ❌ Database queries (data is static)
- ❌ SSR / Next.js / server-side rendering (local webapp only)
