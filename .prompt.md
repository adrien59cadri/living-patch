## Common Patterns to Reference

### Adding a component
Ask Claude/Copilot to create a new component in `src/components/` following the pattern of existing ones (e.g., RelationshipTile). Include Props interface, Tailwind styling, and clear exports.

### Modifying filters
Changes usually go in `src/lib/filters.ts` (filterSpecies function, getFilterOptions) and `src/components/FilterPanel.tsx` (UI). Update label helpers in `src/lib/labels.ts` if adding new tag types.

### Debugging relationships
Check `lib/relationships.ts` (getRelatedEntries, groupByRole) and verify symbiosis data in the JSON dataset. Use browser DevTools to inspect Map objects: `window.__data` or similar if exported.

### Styling issues
Look at existing components (e.g., SpeciesCard, SpeciesList) for TailwindCSS patterns. Color palette: stone (neutrals), emerald (primary), amber (keystone). Breakpoints: `sm:`, `lg:`.

### Type errors
All types live in `src/types/index.ts`. Import Species, Symbiosis, etc. from there. Use `as const` for literal unions. Avoid `any`.

## Before You Ask

1. Check if similar code already exists in the codebase (search SpeciesList for filter pattern, RelationshipsPanel for grouping, etc.)
2. Review MEMORY.md for conventions (component names, file structure, styling approach)
3. Verify the dataset structure if working with data (check app/src/data/dataset.json)
4. Clear your browser cache/hard refresh if changes don't appear
5. Run `npm run dev` and check console for errors

## Testing Strategy

When asking for help with bugs or features, specify what tests you want (or let agent choose):

**For debugging**: "Run only unit tests for [ComponentName]" or "Skip e2e tests and just run unit tests"
- Unit tests are fast (seconds) and isolate logic
- E2E tests are slower (minutes) and valuable only for cross-component workflows
- Default: Agent runs unit test matching the modified file, then e2e only if needed

**For implementation**: "Add tests for this new feature" or "Add/update tests as needed"
- Agent will create test file if none exists
- E2E tests only for user-facing workflows (search, filter, navigation)
- Unit tests for components, library functions, logic

### Example Test Selection

**Debugging FilterPanel**:
```
Task: Debug FilterPanel not showing habitat filter chips
Specs: Modify only FilterPanel.tsx → Run:
  1. npx vitest run src/__tests__/FilterPanel.test.tsx (if exists)
  2. Skip e2e unless full filter workflow seems broken
```

**Debugging search functionality**:
```
Task: Search doesn't find species by latin name
Specs: May affect SpeciesList + search logic → Run:
  1. npx vitest run src/__tests__/SpeciesList.test.tsx
  2. npx playwright test app/e2e/species-list.spec.ts (full workflow)
```

**Adding a new component**:
```
Task: Add new SpeciesPhoto component
Specs: New component, no cross-component dependencies → Run:
  1. npx vitest run src/__tests__/SpeciesPhoto.test.tsx (create this test)
  2. Skip e2e unless SpeciesPhoto is used in detail page
```

See `.instructions.md` → "Testing & Debugging Strategy" for full command reference.

## After You Get Help

1. Agent will run targeted tests during implementation (unit first, e2e if needed)
2. Review test results — fix any failures before moving forward
3. Run the dev server (`npm run dev`)
4. Test the change in browser (search, filter, navigate, go offline)
5. Check TypeScript (`npx tsc --noEmit` from app/)
6. Commit with clear message: `git commit -m "feat: [brief description]"`
7. Update TODO.md if task was tracked

### Test Expectations
- **After small component fix**: Unit tests should pass (seconds to run)
- **After feature implementation**: Both unit and e2e tests should pass
- **During active debugging**: Agent runs only necessary tests for faster feedback loop
- **Coverage report**: Use `npm run test:coverage` if you need visibility into untested code paths
