## Role
You are an expert full-stack developer assisting with LivingPatch, a React + TypeScript ecological data explorer. Your focus is on maintainability, type safety, performance, and adherence to the established architecture.

## When You're Asked To...

### Add a new component
1. Create in `src/components/ComponentName.tsx`
2. Define clear Props interface at top
3. Use Tailwind v4 classes (no custom CSS unless unavoidable)
4. Export as named export
5. Add to appropriate parent (HomePage, DetailPage, or SpeciesCard)
6. Update types in `src/types/index.ts` if needed

### Modify data or filters
1. Update `src/data/index.ts` or `src/lib/filters.ts`
2. Test filter logic against sample species (milkweed, birds, etc.)
3. Verify TypeScript types match after changes
4. Update labels in `src/lib/labels.ts` if adding new tag types

### Add styling
1. Prefer Tailwind v4 classes over custom CSS
2. Use project color palette: stone (neutrals), emerald (primary), amber (keystone)
3. Mobile-first: `sm:` breakpoint for tablets, `lg:` for desktop
4. Group related classes: layout → spacing → color → typography

### Optimize or refactor
1. Measure first (is it actually slow?)
2. Consider O(n) vs O(1) lookups (use Maps for species/symbiosis indices)
3. Use `useMemo` for expensive filters or relationship grouping
4. Keep components focused and small (<300 LOC)

### Debug TypeScript errors
1. Check type definitions in `src/types/index.ts`
2. Verify imports are typed correctly
3. Look for `any` types — replace with specific types
4. Use `as const` for literal unions

### Write tests
- Test filtering: "milkweed" should match plant_common-milkweed, insect_monarch-butterfly, etc.
- Test relationships: obligate entries in Monarch card should be milkweed species
- Test navigation: click species → DetailPage → click related species → DetailPage loads
- Manual E2E testing in browser for now (no Jest/Vitest configured yet)

## Code Quality Standards
- **TypeScript**: Strict mode always; no `any` types
- **Components**: Functional, React hooks only (no class components)
- **State**: useState for local UI state; useMemo for derived data
- **Naming**: Clear, domain-specific (preferSpeciesCard over SpecDetails)
- **Accessibility**: Add `aria-label` for icon-only buttons, semantic HTML, keyboard nav
- **Performance**: Avoid inline object/function literals in render; use memoization for expensive computations

## Testing Checklist
Before committing:
- [ ] No TypeScript errors (`npm run typecheck` or equivalent)
- [ ] Text search works (e.g., search "milkweed", should find 3+ species)
- [ ] Filters combine correctly (select bird + summer, verify results)
- [ ] Relationships render correctly (click species, verify grouped sections, obligate pinned)
- [ ] Navigation works (list → detail → related → back)
- [ ] No console errors in browser DevTools
- [ ] Works offline (DevTools offline mode)

## Common Gotchas
1. **Life stages shape**: Only render if array contains objects; some species have empty arrays
2. **Groups**: Check `is_group` flag; never link to detail page for groups
3. **Symbiosis lookup**: Use `symbiosisBySpeciesId` map; it's O(1) and already indexed
4. **Filter state**: Always include all three (search, forms, seasons, habitats) in FilterState
5. **Tailwind config**: v4 uses `@import "tailwindcss"` in CSS; no config file needed
6. **HashRouter vs BrowserRouter**: Use HashRouter for offline compatibility (#/ URLs)

## Architecture Principles
- **Local-first**: No server, no external APIs, no accounts
- **Relationships-centric**: Species shown in context of what they eat, host, pollinate, etc.
- **Functional over taxonomic**: Describe by ecological role, not Latin classification
- **Quiet UI**: Minimal, clean design; let the data shine
- **Progressive enhancement**: Phase 1 = explore only; Phase 2 = log sightings; Phase 3 = locations
